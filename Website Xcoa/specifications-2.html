Requirements and tests

 On this section we explore the technical requirements identified for our platform to work as intended.

Main Requirements

developer_board

Customed web portal per user

The platform should only show on the web portal the metrics allowed to that user (e.g. If a user does not have permission to manage any temperature sensor, the temperature sensor buttons and pages should not appear.)

view_headline

Sensors types and subtypes

The platform will have the following types and subtypes of streams send by the sensors:
  • Air (CO2 and Pressure)
  • Temperature
  • Waste (Waste container fullness percentage, waste container internal temperature and waste container volume)
  • Noise values
  • Public Lighting (light intensity values and dimming percentage actuator)
  • People counter (percentage of people in a closed space)
  • Radiation (UV radiation, Visible Radiation and Infrared Radiation)

lock

User authentication

The web portal and admin Portal should require user authentication

alarm_on

Fast triggered alarms

When an alarm defined by the user is triggered, if an actuator is associated, the IoT Platform should trigger the actuator in less that one minute

Moving sensors

The system must handle sensors with static or dynamic location

Technical requirements


  • The platform should only show on the web portal the metrics allowed to that user (e.g. If a user does not have permission to manage any temperature sensor, the temperature sensor buttons should not appear.);
  • The platform needs at least one server with public access to be provided by Altice Labs or IT;
  • The platform will have the following types and subtypes of streams send by the sensors: Air (CO2 and Pressure), Temperature, Radiation (UV radiation, Visible Radiation and Infrared Radiation), Waste (Waste container fullness percentage, waste container internal temperature and waste container volume), Noise values, Public Lighting (light intensity values and dimming percentage actuator) and people counter (percentage of people in a closed space);
  • The web portal and admin Portal should require user authentication;
  • Passwords should not be stored in the database, or should be encrypted;
  • The web portal and admin portal must be fully functional in Google Chrome (version 41 or superior), Firefox (version 14 or superior) and Safari (version 7 or superior) and Edge (version 25 or superior) browsers;
  • The mobile application should be compatible with Android 4.0 or superior;
  • The IoT City web server should have a public IP, and be addressable in port 80;
  • The system should handle, at least, 100 online users in the web portal and admin portal simultaneously;
  • The system should handle, at least, 100 mobile users simultaneously;
  • The system should handle, at least, 50 sensors of any type simultaneously;
  • When an alarm defined by the user is triggered, if an actuator is associated, the IoT Platform should trigger the actuator in less that one minute;
  • The mobile app should be self-adjustable to the screen size (mobile and tablet regular screens);
  • The notes and alerts defined by each user should not appear for other user rather that the creator of the note or alert;
  • A user can only see the streams associated to the allowed groups;
  • A user can only see the sensors that have streams associated to the allowed groups;
  • The system must handle sensors with static or dynamic location.

Functional tests

On this section we explain the functional tests of IoT City system using Gherkin syntax.


Web Portal

Title Check the values from a sensor
Narrative As a city manager,
I want to check the values from a sensor,
So that I can understand what is happening in the city.
Acceptance Criteria Scenario 1: City manager sees all streams from a sensor
 Given an authenticated city manager,
And a sensor,
When the city manager selects the sensor,
Then the system shows the information about all streams of the selected sensor.
Scenario 2: City manager toggled a specific stream from a sensor
 Given an authenticated city manager,
And a sensor,
When the city manager selects the sensor,
And the stream,
Then the system shows the information about all streams of the selected sensor.
Scenario 3: Server connection failed
 Given an authenticated city manager,
When the city manager tries to check the values from a sensor,
Then the system shows an error message indicating that there is no connection.

Title Send a value to an actuator
Narrative As a city manager,
I want to send a value to an actuator,
So that I can change its state.
Acceptance Criteria Scenario 1: Value sent successfully
 Given an authenticated city manager,
When the city manager selects the actuator stream,
And the value he wants to send,
Then the system sends that value to that stream.
Scenario 2: Server connection failed
 Given an authenticated city manager,
When the city manager tries to send a value to an actuator,
Then the system shows an error message indicating that there is no connection.

Title Check the location of the sensors in a map
Narrative As a city manager,
I want to check the location of the sensors in a map,
So that I can know the locations where there are no sensors nearby.
Acceptance Criteria Scenario 1: User sees location of sensors
 Given an authenticated city manager,
When the city manager selects the map,
Then the system presents a map with the location of all sensors that the user has access to.
Scenario 2: Server connection failed
 Given an authenticated city manager,
When the city manager tries to check the location of the sensors in a map,
Then the system shows an error message indicating that there is no connection.

Admin Portal


Title Add a sensor to the platform
Narrative As an administrator,
I want to add a sensor to the platform,
So that I can gather more data from the city.
Acceptance Criteria Scenario 1: Administrator input correct information
 Given an authenticated administrator,
When the administrator inputs the sensor id,
And inputs the sensor name,
And inputs the date on which the sensor was added,
And inputs the sensor type,
And inputs the sensor localization,
And inputs if its active,
Then the system registers the new sensor.
Scenario 2: Administrator input wrong information
 Given an authenticated administrator,
When the administrator inputs an already exisiting sensor id,
Then the system shows an error message indicating that there was a problem.
Scenario 3: Server connection failed
 Given an authenticated administrator,
When the administrator tries to add a sensor to the platform,
Then the system shows an error message indicating that there is no connection.
