Features, system scenarios and use cases

Features

 On this section, we explore the main features of our applications (Admin Portal, Web Portal and Mobile Application).


 The admin portal has the following main features:
  • Add sensors to the platform;
  • Add streams to the sensors previously added;
  • Add streams to subscription groups;
  • Add users to subscription groups;
  • Create, edit and delete data on the IoT City Data Persistence (database).

 The web portal has the following main features:
  • Check all the sensors' streams values in real-time;
  • Check the history of sensors' streams;
  • Check the location of all the sensors allowed to the user on a map;
  • Send values to actuators (e.g. change the dimming of lighting);
  • Add alerts with actuators to sensors' streams;
  • Add Rules to sensors' streams;
  • Check the users reports;
  • Add notes associated to the user.

 The citizen mobile application has the following main features:
  • Check the sensors' streams values in real-time and history;
  • Check the location of the sensors in real-time;
  • Send reports of problems or occurrences to the city manager.

 The city manager mobile application has the following main features:
  • Check the sensors' streams values in real-time and history;
  • Check the location of the sensors in real-time;
  • Receive notifications about triggered alerts in real-time.

 For the sensor technician, the platform has the following features:
  • API to add sensors dynamically;
  • API to add streams dynamically;
  • Dynamic location of sensors.

Scenarios

 This section will exemplify four possible scenarios of different type of users taking advantage of the IoT City solution for differentiated goals.


Register sensors


 Mr. Peter, a city manager, wants to gather more metrics from its city. For that, he call to a technician to mounts a sensor in a location in the city where there are no sensors nearby. The tecnician previsouly configures the sensor with its information, and the streams associated. When the sensor is already configured, he mounts it and turns it on, and send the gathered values to the IoT Platform.
 On the IoT City Web Portal, the Administrator of the platform needs to add the new stream to the group of users allowed to see its values.



Experience Map of Mr. Peter to Register Sensors.


Sequence Diagram explaining the flow of messages when registering sensors and streams.


Daily monitoring of the city


 Mr. Peter, the city manager, has arrived to work. He wants to check what happened in the city overnight. For that, he authenticates himself into IoT City Web Portal and checks if there are any abnormal values gathered from the sensors or don't have connectivity, new alarms, new sensors or new situations reported by citizens in the city. Then, he checks if all the situations of the sensors are currently working.

 After confirming that everything looks fine, Mr. Peter wants to see the triggered alarms of the subscriptions in his absence, as well as the reports sent by the users. As there are many reports by the users about public illumination failing during night, he talks to the city workers to solve the occurrence.


Experience Map of Mr. Peter doing the daily monitoring of the city


Schedule alarm with actuator


 Mr. Peter, the city manager, wants to schedule an alarm with an actuator for the public illumination, so that every day from 7 pm to 8 am, all the public illumination is turned on. For that, he starts by defining the days that he wants the alarm to be triggered, and associates the action of turning the lights on and off with that alert.


Experience Map of Mr. Peter scheduling an alarm with an actuator.


Report problem


 Mr. John was having troubles sleeping because of the noise in his street. He decides to open the IoT City Mobile App to check if there are any noise sensors nearby. He notices that there are two noise sensors on his street, and he goes to see his noise level history.

 As the noise level has been high for too long, he decides to report the problem to the city manager, hoping that it will be fixed soon. He writes in the IoT City Mobile App his concerns, also adding an audio file for a more complete description.

Use cases

 On this section, we expose the most important use cases on the system, according to the frequency of use.


Web Portal

Use case Visualize city sensors' streams values in real-time
Description The user visualizes the sensors' streams values that are included in the subscription groups that he is inserted, being divided by metrics.
Exceptions If the user has chosen not to see data from a specific sensor (the streams for it are off), the user will not see data from it unless the stream is turned on.
Priority High
Other information A user can choose not to see data from a certain sensor by turning off the toggle button in the map marker or in the sensors table

Use case Visualize city sensors' stream history
Description The user visualize the sensors' streams history, being shown the values gathered since they were added to the platform.
Exceptions If the user doesn't have access to a sensor or has chosen not to see data from a certain sensor by turning off the stream this data will not be available; those sensors will not apear has an option to see the stream's history
Priority High
Other information When a sensor has more than one stream associated the history values are separated by tabs.

Use case Send values to a city actuator sensor
Description The user act on a sensor, changing its state.
Exceptions The sensor doesn't support actioning; the opipion isn't available to the user
Priority Medium
Other information One example of sensors that support action is the lighting sensors, which allow the user to turn the lights on/off and their intensity.

Use case Visualize city sensors' location on a map
Description The user visualizes the sensors' location on a map and the general information about each sensor.
Exceptions
Priority High
Other information The information about the sensor includes its streams, values and alerts


Mobile Application


Use case Send report
Description The user does a report based on faulty sensors or unusual metric values. Besides the text, he can attach images and audio files. When he finishes, he can send it.
Exceptions
Priority Medium
Other information

Admin portal


Use case Add sensor
Description The city administrator adds a sensor by filling a form with the sensor's information and saves it in the platform.
Exceptions SmartIoT doesn't support the sensor type being introduced; the operation fails
Priority High
Other information The sensors must be compatible with the service to be successfully added


Other Use Cases


Admin portal


Use case Add stream
Description The city administrator add a stream by filling a form with the stream's information and saves it in the plataform.
Exceptions The stream's information provided is not valid; the operation fails
Priority High
Other information The validity of the stream's information filed by the administrator is the responsability of the administrator himself

Use case Add stream to sensor
Description The city administrator add a stream to a sensor by associating the stream to the choosen sensor and saves the changes made.
Exceptions The stream is not compatible with the sensor it is being associated with; an error message is displayed and de operation fails
Priority High
Other information In order to receive data from a sensor a stream has to associated with it.

Use case Create subscription group
Description The city administrator creates a subscription group for a specific group of users that use the web portal.
Exceptions
Priority High
Other information Subscription groups are what determines what each regular user has access to on the platform.

Use case Add sensor to subscription group
Description The city administrator adds a sensor to a specific subscription group and saves the changes made.
Exceptions The subscrition group already has that sensor added; that information is relayed to the administrator and the operation is not executed.
Priority High
Other information Groups of sensor can be added to a stream simultaneously

Use case Add user to subscription group
Description The city administrator adds a user to a subscription group and saves the changes made.
Exceptions The subscrition group already has that user added; that information is relayed to the administrator and the operation is not executed.
Priority Medium
Other information Subscription groups are what determines what each regular user has access to on the platform.


Web portal


Use case Create an alert
Description The user creates an alert of a specific sensor's stream by filling a form for that and save it in the plataform.
Exceptions
Priority Medium
Other information The triggered alerts can be later seen in the dashboard view.

Use case Visualize user reports
Description The user can visualize the user reports, being divided by type of sensor, setting its state of progress.
Exceptions
Priority Medium
Other information

Use case Create a note
Description The user can create a note to remember something he wants by filling a form and save it in the plataform.
Exceptions
Priority Low
Other information These notes can be seen in the main dashboard of the platform.


Mobile application


Use case Visualize sensors' streams values in real-time
Description The user can see the sensors' streams values that are allowed for everyone, being divided by metrics.
Exceptions The user doesn't have access to a specific stream; its data isn't shown.
Priority High
Other information In the dashboard, the values being show for each type are for the closest sensor to the user.

Use case Visualize sensors' location on a map
Description The user visualizes the sensors' location on a map and the general information about each sensor.
Exceptions The user doesn't have access to a specific stream; that sensor's marker isn't shown.
Priority High
Other information The map by default centers in the users location.
